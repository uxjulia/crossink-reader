## Bug 1: JPEGDEC fallback `open failed (err=-1)`

### Root cause (layered)
1. **First attempt** (prior session): The fallback used `s_jpegDecFallbackFile` — a global pointer to the existing `FsFile`. `jpegDecBmpFileOpen` called `seekSet(0)` on it after picojpeg had partially read it. This was unreliable.

2. **Second attempt** (this session, initial): Rewrote `jpegDecBmpFileOpen` to open a fresh `FsFile*` via `Storage.openFileForRead(path)` — same pattern as `JpegToFramebufferConverter`. However, SdFat (on real hardware) cannot open a second read handle while the first (`coverJpg`) is still open on the same file.

3. **Final fix**: Close `jpegFile` inside `jpegFileToBmpStreamInternal` before calling the fallback, since picojpeg has already failed and the file is no longer needed:
   ```cpp
   // lib/JpegToBmpConverter/JpegToBmpConverter.cpp:~line 455
   jpegFile.close();  // Release so JPEGDEC can open a fresh handle
   return jpegFileToBmpStreamViaJpegDec(filePath, bmpOut, targetWidth, targetHeight, oneBit);
   ```

### API change: `filePath` parameter added to all public methods
All three public methods in `JpegToBmpConverter` now require a `const char* filePath` parameter (used so the JPEGDEC open callback can reopen the file by path):

```cpp
// lib/JpegToBmpConverter/JpegToBmpConverter.h
static bool jpegFileToBmpStream(FsFile& jpegFile, Print& bmpOut, const char* filePath, bool crop = true);
static bool jpegFileToBmpStreamWithSize(FsFile& jpegFile, Print& bmpOut, const char* filePath, int targetMaxWidth, int targetMaxHeight);
static bool jpegFileTo1BitBmpStreamWithSize(FsFile& jpegFile, Print& bmpOut, const char* filePath, int targetMaxWidth, int targetMaxHeight);
```

### Callers updated
- `lib/Epub/Epub.cpp:561` — `generateCoverBmp`: passes `coverJpgTempPath.c_str()`
- `lib/Epub/Epub.cpp:652` — `generateThumbBmp`: passes `coverJpgTempPath.c_str()`
- `lib/Txt/Txt.cpp:146` — `generateCoverBmp`: passes `coverImagePath.c_str()`

### Simulator note
The simulator's `JPEGDEC.h` is a stub that always returns `open() = 0` (failure). This is by design — the simulator never actually decodes JPEGs. The error log `JPEGDEC fallback: open failed (err=-1)` in the simulator is expected and harmless. The real hardware uses the actual JPEGDEC library.

## Bug 2: Infinite sleep loop in the simulator

### Root cause
`enterDeepSleep()` calls `esp_deep_sleep_start()` on real hardware, which never returns. In the simulator, the deep sleep stub is a no-op and returns immediately. The auto-sleep check (`millis() - lastActivityTime >= sleepTimeoutMs`) was still true on the next loop iteration, causing an instant re-trigger.

### Fix
`src/main.cpp` — reset `lastActivityTime = millis()` after both `enterDeepSleep()` call sites (auto-sleep and power-button-sleep). This is dead code on real hardware:

```cpp
enterDeepSleep();
// This should never be hit as `enterDeepSleep` calls esp_deep_sleep_start
lastActivityTime = millis();  // Simulator guard: prevents instant re-trigger
return;
```

## Feature: Timezone Setting

**New fields** (`src/CrossPointSettings.h`):
```cpp
static constexpr uint8_t TIMEZONE_UTC0_INDEX = 12;
uint8_t timezoneIndex = TIMEZONE_UTC0_INDEX;  // 0=UTC-12, 12=UTC+0, 26=UTC+14
```

**`TimeStore::applyTimezone()`** (`src/TimeStore.cpp`):
```cpp
void TimeStore::applyTimezone() {
  // POSIX TZ offset sign is inverted from UTC offset
  const int posixOffset = 12 - static_cast<int>(SETTINGS.timezoneIndex);
  char tz[12];
  snprintf(tz, sizeof(tz), "UTC%d", posixOffset);
  setenv("TZ", tz, 1);
  tzset();
}
```
- Called from `main.cpp` after `SETTINGS.loadFromFile()`
- Called from `SettingsActivity.cpp` after every `SETTINGS.saveToFile()` → timezone takes effect immediately without reboot

**Setting** (`src/SettingsList.h`): System category, 27-entry enum:
- `STR_UTC_M12` through `STR_UTC_P14` (all added to `english.yaml`)

**POSIX TZ note**: POSIX TZ sign is **inverted** from ISO 8601. `"UTC-1"` = 1 hour EAST (UTC+1). Formula: `posixOffset = 12 - timezoneIndex`.

Only whole-hour offsets supported. Half-hour zones (India +5:30, Nepal +5:45, Iran +3:30) are not supported.

## Key Architecture Notes

### `JpegToBmpConverter` JPEGDEC fallback pattern
Matches `JpegToFramebufferConverter`'s approach exactly:
- `jpegDecBmpFileOpen(filename, size)`: allocates `new FsFile`, calls `Storage.openFileForRead(filename, *f)`, returns `f`
- `jpegDecBmpFileClose(handle)`: calls `f->close(); delete f;`
- Read/seek callbacks use `pFile->fHandle` (the `FsFile*`)
- The existing `jpegFile` MUST be closed before calling `jpeg->open()` to avoid SdFat refusing a second reader on the same path

### Simulator vs. real hardware
- `HalStorage` on device: uses SdFat with mutex (`StorageLock`); one reader per file at a time
- `HalStorage` in simulator: uses POSIX `::open()` with `./fs_` prefix on all paths; allows multiple readers
- `JPEGDEC` on device: full library, handles progressive and baseline JPEGs
- `JPEGDEC` in simulator: stub, always returns failure (open=0, getLastError=-1, decode=0)
